# IRIS — backend (Node.js + Express + MySQL)

ეს არის ის ნაწილი, რომელიც საიტს ნამდვილად უსაფრთხოს ხდის. `iris-online-store.html` მხოლოდ ვიტრინაა —
ის აჩვენებს მონაცემებს, მაგრამ არაფერს არ წყვეტს ფასზე, მარაგზე ან შეკვეთის ჯამზე. ეს ყველაფერი
ხდება მხოლოდ აქ, სერვერზე, სადაც მომხმარებელს (ბრაუზერს) წვდომა არ აქვს.

## რატომ არის საჭირო

ერთი HTML ფაილი, რომელიც იხსნება პირდაპირ ბრაუზერში, ვერასდროს იქნება რეალურად დაცული — რადგან
ყველაფერი, რაც ბრაუზერშია, მომხმარებელს (ან ნებისმიერს, ვინც დეველოპერულ კონსოლს გახსნის) შეუძლია
შეცვალოს: ფასი, მარაგი, თუნდაც "ადმინად შესვლა". ამიტომ საჭიროა ცალკე სერვერი, რომელიც:

- ინახავს რეალურ მონაცემებს (MySQL-ში, არა ბრაუზერში)
- ყოველი შეკვეთისას თავად ითვლის ფასს და ჯამს ბაზიდან — არ ენდობა იმას, რასაც საიტი გამოგზავნის
- ამოწმებს მარაგს და არ უშვებს შეკვეთას, თუ ნივთი აღარ არის საკმარისი რაოდენობით
- ადმინის პაროლს ინახავს დაშიფრულად (bcrypt) და გასცემს დროებით ტოკენს (JWT) — არა უბრალო "თუ პაროლი სწორია, გახსენი გვერდი" ლოგიკას ბრაუზერში

## შეკვეთის სტატუსები და ავტომატური გასუფთავება

ადმინის პანელიდან შეკვეთას შეგიძლია მიანიჭო ერთ-ერთი ოთხი სტატუსი: **მოლოდინში**,
**დადასტურებული**, **დასრულებული** ან **გაუქმებული**. სერვერი ამოწმებს, რომ მხოლოდ ეს
ოთხი მნიშვნელობა დაიწეროს ბაზაში — ნებისმიერი სხვა მოთხოვნა `/api/admin/orders/:id`-ზე
უარყოფილი იქნება.

`orders` ცხრილს აქვს `updated_at` სვეტი, რომელიც ავტომატურად ახლდება ყოველი სტატუსის
ცვლილებისას. სერვერი ყოველ 24 საათში (და გაშვებისთანავე) ამოწმებს და შლის ყველა
შეკვეთას, რომელიც უკვე 30 დღეზე მეტია **დასრულებული** ან **გაუქმებული** სტატუსშია —
ეს დაცული `FINAL_STATUSES` და `CLEANUP_AFTER_DAYS` კონსტანტებით არეგულირდება `server.js`-ში
(თუ გინდა სხვა ვადა ან სხვა სტატუსების ჩართვა, სწორედ ეს ორი მნიშვნელობა შეცვალე).
გასუფთავება ასევე ხელით გაშვებადია `POST /api/admin/orders/cleanup`-ით (admin token საჭირო).

## დაყენება

1. **MySQL ბაზა:**
   ```
   mysql -u root -p < schema.sql
   ```
2. **გარემოს ცვლადები:**
   ```
   cp .env.example .env
   ```
   შემდეგ `.env`-ში ჩაწერე რეალური MySQL პაროლი, გენერირებული `JWT_SECRET` და შენი ადმინ პაროლის ჰეში:
   ```
   node hash-password.js "შენი-რეალური-ადმინ-პაროლი"
   ```
   მიღებული ჰეში ჩასვი `ADMIN_PASSWORD_HASH`-ში.
3. **დამოკიდებულებები და გაშვება:**
   ```
   npm install
   npm start
   ```
   სერვერი ავტომატურად აეშვება `http://localhost:3001`-ზე (ან `PORT`-ზე, რაც `.env`-შია).

## Frontend-თან დაკავშირება

`iris-online-store.html`-ში, `<script>` სექციის თავში არის ხაზი:
```js
const API_BASE = '';
```
თუ backend სხვა დომენზეა (მაგ. `api.yourdomain.ge`), ჩაწერე:
```js
const API_BASE = 'https://api.yourdomain.ge';
```
თუ იგივე დომენზეა (მაგ. საიტიც და API-იც `yourdomain.ge`-ზეა, უბრალოდ სხვადასხვა route-ზე),
დატოვე ცარიელი — მოთხოვნები წავა `/api/...`-ზე, ამიტომ ამ სერვერიც იმავე დომენზე უნდა იყოს
პროქსირებული (მაგ. Nginx-ით `/api` → `localhost:3001`).

**თუ backend არ არის გაშვებული ან მიუწვდომელია**, საიტი ავტომატურად გადადის "დემო რეჟიმში" —
მუშაობს ბრაუზერის localStorage-ით, მხოლოდ დიზაინის სანახავად. ეს არ არის დაცული და არც სხვადასხვა
მოწყობილობებს შორის გაზიარებული — რეალურ გამოსაყენებლად აუცილებელია ეს backend გაუშვა.

## დეპლოი

მარტივი ვარიანტებია: **Railway**, **Render**, ან ნებისმიერი VPS Node.js + MySQL-ით (მაგ. DigitalOcean).
ბაზისთვის შეგიძლია გამოიყენო მართული MySQL (PlanetScale, Railway MySQL და ა.შ.) ან საკუთარი სერვერი.

## შემდეგი ნაბიჯები რეალურ გამოსაყენებლად

- **გადახდები**: ეს backend არ არის დაკავშირებული არცერთ გადამხდელო სისტემასთან (TBC/BOG/Keepz).
  ამჟამად `payment` მხოლოდ ტექსტური ველია ("ბარათით ონლაინ" / "ნაღდი"), რეალურად თანხა არავისგან
  არ ჩამოიჭრება. საჭიროა შესაბამისი ბანკის/პროვაიდერის API-ის ინტეგრაცია checkout-ის შემდეგ საფეხურად.
- **SMS/Email შეტყობინებები**: შეკვეთის დადასტურების შემდეგ მომხმარებელს არაფერი ეგზავნება — საჭიროა
  SMS API (მაგ. ლოკალური პროვაიდერი) ან email სერვისი.
- **HTTPS**: production-ში backend აუცილებლად უნდა იყოს HTTPS-ის მიღმა (Nginx + Let's Encrypt, ან
  Railway/Render, რომლებიც ამას თავად აკეთებენ).
